## Context

The Medici prediction system now has a **Price Override Queue** backend — SQLite-based job queue where override requests are queued, then picked up by an external `price-override` skill that pushes prices to the Zenith trading platform.

The backend is COMPLETE and TESTED (38 unit tests, all passing). What's missing is the UI integration: override buttons, forms, and status displays woven into the existing dashboards.

### API Endpoints: `src/api/routers/analytics_router.py`
All 7 endpoints exist. Do NOT add new endpoints.

```
POST /api/v1/salesoffice/override/request    → Queue single override
  Body: { "detail_id": 12345, "discount_usd": 1.0 }

POST /api/v1/salesoffice/override/bulk       → Queue all PUT signals
  Body: { "discount_usd": 2.0, "hotel_id": 66814 }
```

### 1. Per-Row Override Button on Options Table

**Where**: The Options HTML page (`/options`) — generated by `src/analytics/options_page.py` or the Jinja2 template it uses.

**What**: Add an "Override" action button to every row that has a PUT or STRONG_PUT signal.

**Behavior**:
- Each PUT/STRONG_PUT row gets a small button: `[⬇ Override]`
- Clicking opens an inline form (no modal — keep it fast):
  ```
  Discount: [$___] (default: $1.00, max: $10.00)
  Target: $249.00 (computed live: current_price - discount)
  [✓ Queue Override] [✗ Cancel]
  ```
- On submit: `POST /api/v1/salesoffice/override/request` with `{ "detail_id": <row_detail_id>, "discount_usd": <entered_value> }`
- On success: show green toast "Queued #ID: $250→$249 for detail 12345", disable button and replace with status badge: `🕐 Pending`
- On error: show red toast with the validation error message
- Non-PUT rows (CALL, NEUTRAL, NONE): no override button, or show it grayed out with tooltip "Only PUT/STRONG_PUT signals can be overridden"

# PART 2: Insert Opportunity (CALL signals) — UI Integration

## Context

The prediction system also has an **Opportunity Queue** backend — SQLite-based job queue for CALL signals. When the system predicts a price will rise by $50+, an operator can queue a purchase opportunity. The external `insert-opp` skill creates BackOfficeOPT + MED_Opportunities records in Azure SQL, and the BuyRoom WebJob executes the actual purchase.

**Key rule**: push_price = buy_price + $50 (fixed). Only CALL options where predicted_price - buy_price >= $50 qualify.

The backend is COMPLETE and TESTED (40 unit tests, all passing).

## Backend Already Built — Do NOT Modify

### Module: `src/analytics/opportunity_queue.py`
Already done. Do NOT change this file.

### API Endpoints (all exist in `analytics_router.py`)

```
  Body: { "detail_id": 12345, "max_rooms": 1 }
  (buy_price and predicted_price come from cached analysis)
  (push_price computed server-side: buy + $50)

POST /api/v1/salesoffice/opportunity/bulk       → Queue all eligible CALLs
  Body: { "max_rooms": 1, "hotel_id": 66814 }

GET  /api/v1/salesoffice/opportunity/queue      → View queue
  Query: ?status=pending&hotel_id=X&batch_id=Y&limit=50&offset=0

GET  /api/v1/salesoffice/opportunity/queue/{id}  → Single request status

GET  /api/v1/salesoffice/opportunity/pending     → External skill picks up work

POST /api/v1/salesoffice/opportunity/{id}/complete → Skill reports result
  Body: { "status": "done", "opp_id": 3866 }
```
